# 第四章：CLI Agent — 从协作到委托 (5-6 页)

> **叙事定位**：同属"工具调用 Agent"阶段（与第三章并列），但形态不同。CLI Agent 脱离 GUI，更适合自动化和批量任务。这一章不加新理论，作为第五章"Computer Use 大跃迁"的铺垫。

## 4.1 为什么需要命令行 AI？

- IDE 很好，但有局限：
  - GUI 占用注意力和屏幕空间
  - 难以集成到 CI/CD 管道
  - 不适合批量、无人值守的任务
- CLI 的核心优势：
  - **可脚本化**：写进 shell 脚本，定时执行
  - **可管道化**：`cat data.csv | claude "分析这个数据"`
  - **可无人值守**：headless 模式，挂后台跑
  - **低资源开销**：不需要 Electron，终端即可
- 类比：IDE AI 是你的"驾驶辅助"，CLI Agent 是你的"自动驾驶"

## 4.2 三大 CLI Agent 对比

| | Claude Code | Codex (OpenAI) | Gemini CLI |
|---|---|---|---|
| **核心能力** | 终端全栈 Agent，读写文件、执行命令 | 沙盒执行、异步任务、Git 集成 | Google 生态、多模态、MCP 扩展 |
| **使用模式** | 交互式 / 管道 / headless | 交互式 / headless / PR 生成 | 交互式 / 管道 |
| **擅长** | 大型代码库理解与重构 | 自动化 PR、代码审查 | 快速脚本、文档处理、系统运维 |
| **生态** | MCP 工具扩展 | GitHub 深度集成 | Google Workspace 联动 |
| **安全模型** | 权限确认 + 白名单 | 沙盒隔离 | 权限分级 |

### Claude Code 重点介绍
- 演示命令：`claude "在这个项目里找到所有未添加单测的函数，生成测试文件"`
- 它会：扫描项目 → 分析函数签名 → 生成测试 → 执行验证 → 报告结果
- 这就是一个完整的闭环——但工具范围限定在**终端能做的事**

### Codex 重点介绍
- 强项：给一个 GitHub Issue，自动生成 PR
- `codex "fix issue #42"` → 读 issue → 分析代码 → 提交 PR → 等你 review
- 适合团队协作场景

### Gemini CLI 重点介绍
- Google 生态联动：可以直接操作 Google Docs、Sheets
- MCP 扩展：可以外挂任意工具（数据库、API、本地脚本）
- `gemini "把这个实验 log 文件整理成 Google Sheet，并生成趋势图"`

## 4.3 实战演示：用 CLI Agent 批量处理实验数据

- **场景**：一个文件夹里有各种格式的实验数据文件（CSV/JSON/TXT），需要：
  1. 统一格式为 CSV
  2. 提取关键指标（均值、方差、异常点）
  3. 生成一份 Markdown 汇总报告带图表
- **演示流程**（用 Claude Code）：
  ```bash
  claude "data 文件夹里有我的实验数据，格式不统一。
  帮我：1) 全部转为 CSV  2) 提取关键统计量  3) 生成 report.md"
  ```
- **演示要点**：
  - 展示 Agent 的**自主决策过程**：它会先 `ls` 看文件、`head` 看内容、选择工具
  - 遇到异常数据时的处理策略
  - 最终输出的报告质量

## 4.4 Skill 机制 — Agent 的"说明书"

> CLI Agent 不是万能的，它需要"学"会你的项目规范。Skill 就是你写给 AI 的标准化说明书。

- **什么是 Skill？**
  - 一份 Markdown 文件，描述特定任务的执行步骤、工具使用规范、约束条件
  - Agent 在执行前会先读取 Skill → 了解"该怎么做" → 再执行
  - 类比：你给新来的实习生写的 SOP 文档

- **Skill 的价值**：
  - 降低信道噪声（回扣信息论）：把隐含知识显式化
  - 提高闭环质量（回扣控制论）：给 Agent 明确的验证标准
  - 减少系统熵（预告系统论）：标准化流程 → 确定性增加

- **实际例子**：
  ```markdown
  # Skill: 实验数据分析
  ## 步骤
  1. 先用 `head -5` 查看文件格式
  2. 如果是 CSV，用 pandas 读取；如果是 JSON，先转换
  3. 必须检查缺失值和异常值
  4. 输出报告必须包含：样本数、均值、标准差、箱线图
  ## 约束
  - 不要删除原始文件
  - 图表必须保存为 PNG
  ```

## 4.5 工具扩展：MCP vs CLI — 一个前瞻性判断

> 这一节提出一个有争议但有深度的观点，适合引发思考。

### 什么是 MCP（Model Context Protocol）？
- Anthropic 提出的标准化协议：让 AI 通过统一接口调用外部工具
- 类比：USB 接口——任何设备只要遵循协议就能即插即用
- 目前各大 AI 工具都在接入 MCP（Gemini CLI、Claude Code 等）

### CLI 工具调用 — 更古老但更可靠
- AI 直接执行命令行工具（`curl`、`python`、`grep`、`ffmpeg`…）
- 无需额外协议，操作系统就是"协议层"
- 几十年的工具生态，覆盖几乎所有场景

### 一个前瞻性判断：MCP 可能会消失，CLI 会长存

```
         MCP 的定位困境
         ─────────────

   ┌─────────────┐
   │  应用 API   │ ← 每个应用有自己的 API
   ├─────────────┤
   │    MCP      │ ← MCP 试图做统一适配层
   ├─────────────┤
   │  AI Agent   │ ← Agent 通过 MCP 调用工具
   └─────────────┘

   问题：当 AI 足够聪明，它可以直接读 API 文档、
   写代码调用 API。MCP 这个中间层的价值在哪？
```

- **为什么 CLI 会长存？**
  - CLI 工具是操作系统的原生语言，不会因为 AI 生态变化而消失
  - AI 天然擅长处理文本 → CLI 的 stdin/stdout 是最高效的 Agent 接口
  - 几十年积累的工具链 → 无需重新发明轮子
  - Skill 文件 + CLI 工具 = 轻量、灵活、可离线、无依赖

- **MCP 的价值窗口**：
  - 目前有用：在 AI 能力不够强时，MCP 降低了工具对接的门槛
  - 长期来看：随着 AI 代码能力增强，Agent 可以自己写胶水代码调用任何 API
  - 企业级场景可能保留：合规、审计、权限管理等需求

- **给学生的思考**：不要把赌注押在某个协议上，理解底层原理（CLI、API、IPC）比追协议更持久

## 4.6 🧠 上下文限制 — Agent 的阿喀琉斯之踵

> 所有 Agent 工具都绕不开的核心问题。这一节为第五章的系统论深潜做铺垫。

### 问题本质
- 上下文窗口 = Agent 的"工作记忆"
- 再大的窗口也是有限的（128K? 200K? 1M?）
- 任务越复杂 → 上下文越多 → 窗口越容易溢出 → 关键信息被截断

### 三种主流解决方案

**方案 A：自动上下文压缩**
- AI 自动总结过长的上下文，保留关键信息，丢弃冗余
- Claude Code / Gemini CLI 都在用
- 信息论视角：这是有损压缩 → 必然丢失信息 → 问题是"丢了什么？"
- 风险：AI 认为不重要的信息，可能恰好是后续步骤的关键

**方案 B：长记忆 Markdown 文件管理（OpenClaw 方案）**
- Agent 把重要上下文写入持久化的 .md 文件
- 每次启动时读取 → 相当于"外部记忆"
- 优势：持久、可审计、人类可编辑
- 类似方案：Claude Code 的 CLAUDE.md、Cursor 的 .cursorrules
- 本质：用**文件系统**扩展上下文窗口

**方案 C：向量数据库索引 + RAG（检索增强生成）**
- 把大量文档切片 → 向量化 → 存入向量数据库（Milvus、Pinecone、ChromaDB）
- Agent 需要信息时 → 语义检索最相关的片段 → 注入上下文
- 优势：可以管理海量文档，按需召回
- RAG 原理简述：Retrieval → Augment prompt → Generate
- 适合场景：大规模知识库、论文库、代码库

### 对比总结

| 方案 | 容量 | 精度 | 成本 | 适合场景 |
|------|------|------|------|---------|
| 自动压缩 | 中 | 中（有损） | 低 | 单任务、短期会话 |
| Markdown 长记忆 | 中 | 高 | 低 | 项目级知识、规范文档 |
| 向量数据库 RAG | 高 | 依赖检索质量 | 中-高 | 大规模知识库、论文库 |

- **核心洞察**：上下文管理是当前 AI 工具链的**核心技术瓶颈**
- 引出系统论：上下文不足 → 信息丢失 → 节点间传递失真 → 系统熵增加

## 4.7 CLI Agent 的能力边界

- CLI Agent 能做的 = **终端能做的一切**
- 但它不能：
  - 操作 GUI 应用（Word、浏览器、Figma）
  - 跨应用工作流（邮件 → 日历 → 文档）
  - 感知屏幕内容
- 用一张能力边界图：

```
┌─────────────────────────────────────────────┐
│              操作系统全部能力                  │
│  ┌────────────────────────────────────────┐  │
│  │         GUI 应用操作                    │  │
│  │  ┌──────────────────────────────────┐  │  │
│  │  │      终端/命令行 能力             │  │  │
│  │  │  ┌────────────────────────────┐  │  │  │
│  │  │  │    纯文本生成              │  │  │  │
```
